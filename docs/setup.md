# Cinderbox Chat — Self-Hosting Guide

## Requirements

- **PHP 8.0 or later** with the following extensions:
  - `pdo` (PDO base)
  - `pdo_mysql` (MySQL driver)
  - `json` (usually bundled)
- **MySQL 8.0+** or a compatible MariaDB version
- Any standard web host — shared hosting works fine
- **HTTPS** is strongly recommended (required for the Service Worker and PWA features)

No Node.js, no Composer, no Docker, no build step.

---

## Deployment

### Step 1: Upload the files

Upload the following files to your web root (e.g., `~/public_html/` or `~/yourdomain.com/`):

```bash
scp api.php index.html sw.js manifest.json icon.svg user@yourhost.com:~/public_html/
```

Only `index.html` and `api.php` are strictly required for the application to function. `sw.js`, `manifest.json`, and `icon.svg` are needed for PWA/offline support and the install prompt.

### Step 2: Visit the site

Open the site in your browser. Because `config.php` does not exist yet, the API returns `503 setup_required` and the frontend automatically displays the setup screen.

### Step 3: Enter your database credentials

Fill in the setup form:

| Field | Description |
|-------|-------------|
| MySQL Host | Hostname of your MySQL server (e.g., `localhost` or `db.example.com`) |
| Port | MySQL port (default: `3306`) |
| Database Name | The name of an existing database the user has access to |
| Username | MySQL username |
| Password | MySQL password (may be blank for some local setups) |

### Step 4: Submit

The setup endpoint (`?action=setup`):

1. Tests the database connection using the provided credentials.
2. Writes `config.php` to the same directory as `api.php` with the credentials as PHP `define()` constants.
3. Runs the database migrations (creates all required tables).
4. Returns `{"ok": true}` on success, redirecting the frontend to the chat interface.

If `config.php` cannot be written, the error message "Could not write config.php — check directory permissions" is displayed. Fix the directory permissions and try again.

### Step 5: Verify

After setup, the `?action=setup` endpoint permanently returns `403 already_configured`. There is no way to re-run setup without manually deleting `config.php`.

## Database Schema

The schema is created automatically by the migrations system. You do not need to run any SQL manually.

Tables created:

- **`schema_version`** — tracks which migrations have been applied
- **`rooms`** — one row per active room
- **`messages`** — all pending messages (rows are deleted after delivery)
- **`presence`** — active participants per room

The migration system runs on every request (not just setup). When you deploy a new version of `api.php` that includes new migrations, they are applied automatically on the first request after deployment.

### Migration history

| Version | Change |
|---------|--------|
| 1 | Creates `rooms`, `messages`, `presence` tables |
| 2 | Adds `encryption_test` column to `rooms` |
| 3 | Widens `messages.ciphertext` from TEXT to MEDIUMTEXT (supports larger messages) |

## Updating

To deploy an update:

```bash
scp api.php index.html sw.js manifest.json icon.svg user@yourhost.com:~/public_html/
```

No other steps are required. Any new database migrations run automatically on the first request after the new `api.php` is in place.

`config.php` is never overwritten by this process — it is not included in the deployment and its path is excluded from git via `.gitignore`.

---

## config.php

`config.php` is generated by the setup endpoint and contains:

```php
<?php
define('DB_HOST', 'localhost');
define('DB_PORT', 3306);
define('DB_NAME', 'cinderbox');
define('DB_USER', 'dbuser');
define('DB_PASS', 'dbpassword');
```

**Never commit this file.** It is excluded by `.gitignore`. If it ends up in version control by accident, rotate your database credentials immediately.

The file must be readable by the PHP process (typically the web server user). If `api.php` and `config.php` are in the same directory and the web server can execute `api.php`, it can read `config.php` via `require`.

## Reverse Proxy (nginx example)

If you are running Cinderbox Chat behind nginx as a subdomain, a minimal configuration looks like:

```nginx
server {
    listen 443 ssl;
    server_name chat.example.com;

    ssl_certificate     /etc/letsencrypt/live/chat.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/chat.example.com/privkey.pem;

    root /var/www/chat.example.com;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location ~ \.php$ {
        fastcgi_pass unix:/run/php/php8.2-fpm.sock;
        fastcgi_index index.php;
        include fastcgi_params;
        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
    }

    # Prevent config.php from being served directly
    location = /config.php {
        deny all;
    }
}
```

## Troubleshooting

### Blank page / no UI

The frontend is entirely client-side JavaScript. A blank page usually means the browser blocked script execution or the HTML file is not being served correctly.

- Check that `index.html` is served with `Content-Type: text/html`.
- Open the browser developer console for JavaScript errors.
- Verify that `api.php` is accessible — try visiting `https://yoursite.com/api.php?action=ping` directly. It should return `{"ok":true}`.

### Setup screen loops or "Could not write config.php"

The web server process needs write permission on the directory containing `api.php` in order to create `config.php`.

```bash
# Check ownership
ls -la /path/to/webroot/

# Allow the web server to write (adjust www-data to your server user)
chown www-data:www-data /path/to/webroot/
```

After fixing permissions, try the setup form again.

### Messages not delivering

1. Confirm the MySQL connection is working: check that `config.php` has the correct credentials and that the MySQL server is reachable from the PHP process.
2. Check that the `messages` table was created: connect to the database and run `SHOW TABLES;`.
3. If using a remote MySQL host, ensure the MySQL user has `SELECT`, `INSERT`, `DELETE`, `UPDATE`, and `CREATE` privileges on the database.

### "Room not found or has expired"

Rooms with no activity for more than 7 days are automatically purged. If a room was recently active, the room ID may have been entered incorrectly.

Retention-limited rooms (1h, 6h, 12h, 24h) expire their messages after the retention period, but the room row itself persists until the 7-day abandonment threshold.

### Service Worker not registering / app not installable

The Service Worker API requires a **secure context** (HTTPS or localhost). On plain HTTP, `navigator.serviceWorker` is unavailable and the PWA install prompt will not appear. The chat itself functions over HTTP, but without offline support or installability.

If you are on HTTPS and the Service Worker is still not registering, check that `sw.js` is being served from the same origin as `index.html` (same scheme, host, and port).

## Security Hardening Recommendations

1. **Use HTTPS.** TLS is required for the Service Worker, protects credentials in transit (even though message content is encrypted client-side), and prevents network-level tampering with the HTML/JS files. Use Let's Encrypt for free certificates.

2. **Keep PHP and MySQL up to date.** `api.php` targets PHP 8.0+. Running on an end-of-life PHP version exposes the server to unpatched interpreter vulnerabilities.

3. **Protect config.php from direct HTTP access.** On Apache, add a `.htaccess` rule; on nginx, use a `deny all` location block (see example above). The application will still work because PHP reads it via `require`, not via HTTP.

4. **Consider nginx-level rate limiting** for the sync endpoint, in addition to the application-level rate limit (60 messages per sender tag per minute):

   ```nginx
   limit_req_zone $binary_remote_addr zone=api:10m rate=30r/m;
   location = /api.php {
       limit_req zone=api burst=10 nodelay;
       # ... fastcgi config
   }
   ```

5. **Restrict database privileges.** The MySQL user only needs `SELECT`, `INSERT`, `UPDATE`, `DELETE` on the application tables, plus `CREATE` and `ALTER` for migrations. It does not need `DROP`, `GRANT`, or `FILE` privileges.

6. **Do not expose error output.** `api.php` sets `error_reporting(0)` and disables display/logging of errors. Verify this is effective on your host — some shared hosts override `php.ini` settings. Check that a request to a non-existent action returns `{"error":"unknown_action"}` and not a PHP warning.
